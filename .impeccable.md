## Design Context

This dashboard is the sibling product to **web3insight.ai** (the marketing / profile site) and **must share its visual language**. Treat that site as the canonical reference: open it at https://web3insight.ai/ when in doubt, and mirror its tokens, type system, chrome, and chart palette here rather than inventing new ones. The dashboard is denser and more data-driven than the marketing site, but the design language is one system — not two.

The system is internally named **Blueprint**: a three-ink composition of a tinted-cream ground, near-black ink, and a rare teal trace. Everything below is expressed in that vocabulary.

**Secondary reference: https://sandbox.cloudflare.com/.** Cloudflare's Sandbox SDK site is the clearest public example of the same engineering-blueprint language we want — orthogonal trace overlays across the whole layout, ruler tick strips, corner brackets and corner codes, stacked outlined display wordmarks, code panels with muted syntax tokens, hand-drawn script labels with arrows, and dotted / hatched ground fills. When a chrome decision isn't settled by web3insight.ai alone, look to sandbox.cloudflare.com as a compatible reference dialect and pick whichever reads more "drafting sheet" and less "SaaS dashboard." Do **not** copy their accent hue — ours is teal, not Cloudflare orange.

### Users

Web3Insight serves a mixed but overlapping audience: ecosystem foundations tracking their developer base, grant reviewers evaluating contributors before funding, DAO ops running contributor programs, web3 companies scouting talent, bounty hunters proving their work, and individual developers showcasing on-chain + open-source contributions as a portfolio.

Usage spans day and night — daytime focused review sessions (grant committees, ecosystem reports, hiring decisions) and late-night hackathon / research use. Sessions are data-intensive and often comparative (ranking dev A vs dev B, ecosystem X vs ecosystem Y). The core job-to-be-done is *learning more about a web3 subject* — a dev, a repo, an ecosystem, a cohort — with enough signal density to form a defensible judgment.

**Implication:** the interface must reward long, dense looks without fatigue, and must feel credible to someone writing a grant memo with it open.

### Brand Personality

Three words: **rigorous, on-chain-native, irreverent.**

- **Rigorous** — numbers and methodology are visible, not hidden behind hero charts. Users can trace a metric back to its source (tooltip, footnote, small-caps label like `src: opendigger · 30d`). Nothing is merely decorative.
- **On-chain-native** — addresses, hashes, block heights, repo SHAs, commit ids are first-class citizens, styled as such in a monospace specimen. The interface speaks the audience's vocabulary without explaining it.
- **Irreverent** — not serious in the corporate-finance sense. There is dry wit in empty states, editorial cadence in annotations, and willingness to be opinionated about what matters. Playful through typographic and editorial choices (outlined display stacks, hand-drawn script labels with arrows, revision numbers like `rev · 0001 · sheet 01/04`), *not* through emoji, mascots, or color confetti.

Emotional goal: the user feels *equipped* — like they have the good seat at the table.

### Aesthetic Direction

**Engineering-blueprint crossed with data journalism.** Think the forensic density of Arkham / Dune / Nansen combined with the editorial confidence of the Financial Times' visual-journalism pieces, Pudding long-reads, and Observable notebooks — then drawn on an architect's drafting sheet. Information-rich, annotation-heavy, comfortable with small type, tabular numerics, visible rulers, and hand-labeled callouts.

**Anti-references** (do not drift toward these):
- Generic SaaS analytics templates (the Mixpanel / Amplitude / "kpi-card-grid-with-sparkline" look)
- Neon-purple-gradient Web3 landing slop
- Bloomberg Terminal cosplay (amber-on-black, all-caps headers everywhere)
- Glassmorphic crypto dashboards with floating cards and glow borders
- Anything that would look indistinguishable from another AI-generated dashboard

**Theme.** Light is default — most review and reporting happens during the day and light mode reads as more credible for grant/reporting contexts. Dark is offered for late-night research sessions. Both themes are first-class; neither is a retrofit of the other. Toggled via `[data-theme="dark"]` / the `.dark` class depending on surface (kept in sync through `next-themes`).

**Typography — shared with web3insight.ai; do not substitute without moving both products together.**

- **Editorial display** — `Bricolage Grotesque` (variable, optical sizing). Used for section titles, hero numbers as heroes, and the outlined-display stacks. Letter-spacing `-0.01em`.
- **Body / UI** — `Manrope` on the marketing site; `Host Grotesk` in the dashboard. Both are warm, low-contrast sans families chosen deliberately to avoid the reflex defaults (Inter, Geist, Plex, Syne, Fraunces, Playfair). Treat them as interchangeable at the system level; do not add a third body face.
- **Mono / tabular / on-chain chrome** — `JetBrains Mono`. Reserved for addresses, hashes, numeric columns, terminal-style panels, small-caps eyebrow labels, rulers (`rev · 0001`, `sheet 01/04`), and source annotations. Monospace is a specimen, **not a theme** — do not use it for body prose, paragraph text, headings, or "this looks technical" decoration.
- **Script label** — `Caveat` (only on the marketing site for `HandLabel` callouts like `start →`). Not currently in the dashboard; introduce only if a hand-drawn callout is genuinely warranted.

Body feature-settings `"ss01", "cv01"` on the marketing site and `"ss01", "cv11"` on the dashboard — both enabled. Mono has `"zero"` + `"ss01"` and `font-variant-numeric: tabular-nums` so every digit column aligns.

### Color — OKLCH, three-ink discipline

**Teal `#0F766E` is locked** (logo color). It is the product's sole brand hue and must harmonize the whole system. The full teal ramp is expressed in OKLCH with reduced chroma at the extremes, so the near-white and near-black stops sit comfortably beside the neutrals.

Canonical ramp (from `web3insight.ai/src/app/globals.css`):

```
--teal-50:  oklch(96% 0.018 180)
--teal-100: oklch(90% 0.030 180)
--teal-200: oklch(82% 0.050 180)
--teal-300: oklch(72% 0.070 180)
--teal-400: oklch(58% 0.080 180)
--teal-500: oklch(47% 0.080 180)   ← brand trace / accent
--teal-600: oklch(40% 0.075 180)
--teal-700: oklch(33% 0.065 180)
--teal-800: oklch(26% 0.050 180)
--teal-900: oklch(20% 0.035 180)
--teal-950: oklch(15% 0.020 180)
```

**Neutrals are tinted toward the teal hue** (hue `180`, chroma `0.004–0.010`). No pure grays. No pure `#fff` / `#000`. Light surfaces warm slightly toward teal; dark surfaces cool slightly toward teal.

Semantic mapping (Blueprint naming on the marketing site / dashboard equivalents):

| Role                   | Marketing site variable | Dashboard variable    | Light OKLCH (target)       |
| ---------------------- | ----------------------- | --------------------- | -------------------------- |
| Ground                 | `--background`          | `--bg`                | `oklch(97% 0.004 180)`     |
| Raised surface         | `--card`                | `--bg-raised`         | `oklch(98.5% 0.004 180)`   |
| Sunken / strip         | `--muted`               | `--bg-sunken`         | `oklch(95% 0.005 180)`     |
| Ink (body text)        | `--foreground`          | `--fg`                | `oklch(18% 0.008 180)`     |
| Ink muted              | `--muted-foreground`    | `--fg-muted`          | `oklch(42% 0.010 180)`     |
| Hairline (rule)        | `--border`              | `--rule-strong`       | `oklch(18% 0.008 180)`     |
| Soft hairline          | `--border-soft`         | `--rule`              | `oklch(78% 0.008 180)`     |
| Trace (accent)         | `--accent` / `teal-500` | `--accent`            | `oklch(47% 0.080 180)`     |
| On-trace foreground    | `--accent-foreground`   | `--accent-fg`         | ground color (cream)       |
| Warn / pull-quote      | `--destructive`-adj     | `--warn` (amber-ochre)| `oklch(65% 0.130 60)`      |

Dashboard-specific extras (kept, not expanded): an amber ramp for warning / editorial pull-quote emphasis (`--amber-100/400/600/800`) and a minimal danger pair (`--danger-500/600`, hue 25). **Never** use these together with teal in the same chart gradient.

**Accent discipline.** Teal is the 10% accent. Its power comes from rarity. Use it for: active state, selected row, key number, the single chart series the user is meant to follow, and the trace lines that connect panels. Do not tile it across cards, hero gradients, or 30% of the surface.

**Banned color moves.** Cyan-on-dark. Purple-to-blue gradients. Neon green "terminal" text. Any gradient fill on text. Teal-to-indigo gradients. The legacy `#6366F1` secondary and `gradient-x` / `pulse-glow` / `shimmer` keyframes from the pre-Blueprint config are retired — do not resurrect them.

### Radius, hairlines, and chrome

The Blueprint is **sharp**: `--radius` is `0.125rem` (2px) on the marketing site. In the dashboard that same tight radius is in effect via `--radius-md: 0.125rem` / `--radius-lg: 0.125rem`; the `sm` radius is `0px`. Soft-rounded, pill-shaped, or `rounded-xl` shapes belong to a different product — avoid them.

Hairlines are 1px, in two weights:
- `border` / `--rule-strong` — ink hairline for primary structure (panel outline, header strip).
- `border-soft` / `--rule` — muted hairline for secondary subdivisions (ruler strips, ticks, inline dividers).

### Signature chrome elements (shared primitives)

These live in `web3insight.ai/src/components/blueprint/` on the marketing site. The dashboard should import the same vocabulary, named identically, so the two products read as one:

1. **`Panel`** — bordered surface with optional corner `label` (e.g. `core · devs`) and opposite-corner `code` (`01` / `02` / …), and an optional `ground` fill of `plain`, `dotted` (14px radial dot grid), or `hatched` (7px diagonal lines at 45°). Labels sit on the border line itself (`-translate-y-1/2`, `bg-background px-1.5`) so they read as etched captions, not floating pills.
2. **`Trace`** — SVG overlay that draws orthogonal connector paths between panels (`M … L … L …`), using the `animate-draw` stroke-dashoffset keyframe and optional filled/outlined node circles. Always in teal or ink — never rainbow.
3. **`OutlinedDisplay`** — stacked text-stroke echoes under a solid front layer, used for the hero wordmark and other large editorial displays. Stack of 4–5, offset 4–6px, front layer solid.
4. **`EdgeTicks`** — ruler tick marks along the outer edges of a container (repeating linear-gradient). Decorative chrome only (`aria-hidden`, `pointer-events-none`).
5. **`HandLabel`** — Caveat-script callout with a ± slant and optional directional arrow. Used very sparingly; never in data-dense regions.
6. **`TerminalPanel`** — mono-spaced panel with a small header strip, tokenized syntax (`keyword`, `string`, `comment`, `prompt`, `ident`, `punct`, `plain`) and a blinking block cursor (`animate-cursor`). Replaces "loading dots" and spinner animations in the Blueprint.
7. **`LogoMark`** — 3×3 grid of 6px squares, centre square filled teal, the other eight hairline-outlined. Reads as a schematic / PCB chip emblem.

The dashboard already ships analogous primitives in `src/components/primitives/` (`SectionHeader`, `SmallCapsLabel`, `DenseTable`, `Address`, `Hash`, `NumericCell`, `BigNumber`, `EmptyState`, `AnnotationRail`, `MetaList`). Treat those as the data-surface vocabulary and the Blueprint set above as the chrome / composition vocabulary. Reach for both when building a page.

### Motion

- Entrance: `fade-in-up` — 0.6s, `cubic-bezier(0.16, 1, 0.3, 1)`, 20px Y translation. Stagger children by 80–160ms via `framer-motion` variants.
- Trace drawing: `animate-draw` — `stroke-dashoffset` from `--draw-length` to 0 over 1.4–1.8s with the same ease-out curve; nodes fade in behind the trace at `delay + duration * 0.8`.
- Loading states: a blinking terminal block cursor (`animate-cursor`, 1s step-end, 50/50 duty). **Never** shimmer-loaders on non-async content, glow pulses, bounce eases, or spring overshoot.
- Everything respects `prefers-reduced-motion: reduce`.

### Design Principles

1. **Density is the feature, not a bug.** Design for the review-memo user who wants a lot of signal in one screen. Resist the urge to pad, inflate, or hero-ify. A dense, well-typeset table beats four KPI cards.

2. **Show the methodology.** Every metric reveals its definition on hover or in an adjacent annotation. Credibility comes from visible sourcing — tooltips, footnotes, small-caps labels like `source: opendigger · 30d window`, ruler strips with `rev · 0001 · sheet 01/04`.

3. **Editorial hierarchy over card grids.** Use asymmetric layouts, varied spacing rhythm, left-aligned type with max-width ~65–75ch on prose. Mix hero charts with narrow annotation columns, like a long-read article. Avoid uniform 12-column card grids where each cell is interchangeable.

4. **Monospace is a specimen, not a theme.** Addresses, hashes, repo SHAs, numeric columns, terminal panels, eyebrow labels, source annotations, and ruler ticks get mono. Headings, body prose, and labels do not.

5. **One rare teal, many tinted neutrals.** The interface is built out of teal-tinted off-whites and near-blacks. Teal itself shows up at moments of emphasis: active state, key number, selected row, the one chart line the user is meant to follow, the trace connector.

6. **Motion is reportorial, not performative.** Staggered entrance on page load, exponential ease-out transitions, traces that draw themselves, cursors that blink. No bounce. No glow pulses. No shimmer on static content.

7. **Empty states teach, not apologize.** Written with dry wit and a next action. Example voice: *"No repos under this filter. Try widening the window, or check `ecosystem=starknet` — that one's always busy."*

8. **Charts share one palette.** The same teal-dominant scale feeds ECharts, Highcharts, and Recharts from a single tokens file. Do not let each chart library pick its own colors.

### Implementation Notes (project-specific)

- `src/app/globals.css` is the source of truth for dashboard tokens. Keep the variable names (`--bg`, `--bg-raised`, `--bg-sunken`, `--fg`, `--fg-muted`, `--fg-subtle`, `--rule`, `--rule-strong`, `--accent`, `--accent-fg`, `--accent-muted`, `--accent-subtle`, `--warn`, `--danger`) and keep the OKLCH values in lock-step with `web3insight.ai/src/app/globals.css`. When the marketing site's palette shifts, migrate the dashboard in the same PR.
- The dashboard uses Tailwind via `@tailwind` directives; colors are consumed as CSS variables (`bg-bg`, `text-fg`, `border-rule`, etc.). Do **not** re-introduce hex colors or the legacy Tailwind config hues that the pre-Blueprint `tailwind.config.ts` carried.
- Fonts are wired via `next/font` in `src/app/layout.tsx`: `Bricolage_Grotesque` → `--font-bricolage`, `Host_Grotesk` → `--font-host`, `JetBrains_Mono` → `--font-jetbrains`, exposed as `--font-display` / `--font-sans` / `--font-mono` in `globals.css`. Keep this indirection so marketing and dashboard can independently swap their body face without breaking the semantic names.
- `next-themes` with the `[data-theme="dark"]` selector is the existing wiring. Keep it.
- Charting stack is ECharts + Highcharts + Recharts. A single teal-dominant token set (reading from the CSS variables) must feed all three; never let a chart component hard-code a palette.
- When porting marketing chrome into the dashboard (ruler strips on the header, panel labels on data cards, trace connectors between related panels), import the Blueprint primitives rather than re-implementing them — the contract for `Panel`, `Trace`, `TerminalPanel`, `HandLabel`, `EdgeTicks`, `OutlinedDisplay`, and `LogoMark` is documented in `web3insight.ai/src/components/blueprint/`.
- Ground fills (`.dotted-pattern`, `.diagonal-lines`) and motion utilities (`.animate-draw`, `.animate-fade-in-up`, `.animate-cursor`) must match both projects byte-for-byte. They are part of the design-system contract.
